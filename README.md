# Japanese Handwriting Pad Prototype

GitHub Pages で公開しやすい、静的な手書き認識パッドのプロトタイプです。

## 実装内容

- 手書きパッド
- クリアボタン
- 戻るボタン
- `Z` キーによる直近筆画の取り消し
- 認識された文字の表示ボックス
- 画数を考慮した即時手書き認識
- 複雑な候補の安定性監視
- PC、タブレット、スマホ向けの Pointer Events 対応
- タッチ入力時のダブルタップ拡大、長押し選択の抑制

## 使い方

ローカル確認では、`file://` ではなく簡易サーバー経由で開いてください。

```bash
python3 -m http.server 8080
```

その後、ブラウザで `http://localhost:8080/` を開きます。

GitHub Pages では、このリポジトリのルートにある `index.html` をそのまま公開できます。

## 認識エンジンについて

このプロトタイプは、まずブラウザの Handwriting Recognition API を試し、利用できない環境では Google Input Tools の手書き認識リクエスト形式にフォールバックします。

注意点:

- 手書きパッドへの描画は、認識エンジンの対応状況に関係なく動作します。
- 書き途中の字を完成字として誤表示しにくくするため、認識候補を表示する前に標準画数、物理的な筆画数、つなげ書き推定数を比較します。
- つなげ書きの線は、角度変化から複数の筆画として推定します。
- しんにょうを含む候補は、左下から右へ伸びる最終筆画があるかも確認します。
- さんずい、くさかんむりを含む候補は、それぞれの部品が成立しているかを追加確認します。
- 複雑な候補、部品ガード対象候補、画数不明の漢字候補は、同じ筆跡状態で候補が安定してから表示します。
- `鱸` は外部画数データに含まれないため、補助画数データとして 27 画を追加しています。
- クリックだけでは点を描画せず、マウス・ペンは8px、タッチは10px以上ドラッグしてから筆跡を開始します。
- 短すぎる入力や日本語以外の文字を含む候補は表示しません。
- ブラウザ内蔵APIが日本語に対応している環境では、端末側の認識機能を使います。
- Edge、Chrome、Brave などで内蔵APIが使えない場合は、フォールバックとして外部の認識サービスへストローク情報を送ります。
- フォールバック認識にはネットワーク接続が必要です。

本格運用で日本語の認識精度と対応範囲を固定したい場合は、手書き認識エンジンを別途組み込む設計にするのが安全です。
